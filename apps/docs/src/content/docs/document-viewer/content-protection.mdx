`contentProtection` is an opt-in flag that applies a small bundle of **client-side capture deterrents** to the viewer, aimed at discouraging a casual reader from grabbing or exfiltrating a confidential document.

```vue
<DocumentViewer
  source="/confidential.pdf"
  wasm-url="/pdfium.wasm"
  :features="{ contentProtection: true }"
/>
```

It defaults to `false`, so existing viewers are unaffected.

## Read this first — what "protection" can and can't mean

> **This is a deterrent, not a guarantee.** No web page can stop a determined viewer from capturing what's on their own screen. Everything `contentProtection` does runs in the page's JavaScript and is **trivially removable** — a viewer can open browser DevTools, disable JavaScript, or delete the overlay node and it's all gone. Treat it as friction against casual capture, never as a security control for truly sensitive material.

Specifically, `contentProtection` **does not** stop:

- **OS screenshots** — macOS `Cmd+Shift+3/4`, Windows `Win+Shift+S` snipping, and **Linux `PrintScreen`** (GNOME/KDE grab it as a global hotkey and save straight to `~/Pictures/`). These write to disk without notifying the page — the JS never even sees the key.
- **Screen recorders** — OBS, QuickTime, Loom, the OS recorder, or a screen-share in a meeting.
- **A phone photo** of the monitor.
- **A technical user** with DevTools.

The only browser technology that genuinely makes captures come out black is **DRM-protected media** (EME + Widevine / PlayReady / FairPlay with HDCP), which applies to `<video>`/`<audio>` streams — not arbitrary PDF / canvas / DOM content. That is out of scope for this component. If you have a hard "the viewer must not be able to screenshot this" requirement, a document viewer in a browser is the wrong delivery mechanism.

## What it actually does

When `contentProtection` is `true`, four deterrents are applied at the viewer root (so they cover PDF, image, text, and markdown sources alike):

| Deterrent                 | Behaviour                                                                                                                                | Honest note                                                                                                                                                     |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Obscure on leave**      | When the window loses focus or the tab is hidden, the document is covered by a "Protected content" overlay; it clears when focus returns | The closest thing to screen-record / snipping-tool deterrence. Expect **false positives** — a notification, an alt-tab, or opening DevTools also triggers it    |
| **PrintScreen deterrent** | On the `PrintScreen` key, the clipboard is best-effort cleared and the overlay shows briefly                                             | **Windows / Chromium only.** Does nothing on **Linux** (the desktop environment grabs PrintScreen before the page and saves to a file) or **macOS** (no PrintScreen key). It also **wipes whatever the user had on their clipboard** |
| **Block context menu**    | Right-click is suppressed on the viewer                                                                                                  | Stops the casual "Save image as…" / "Copy image". Not screenshot-related                                                                                        |
| **Block drag-out**        | Dragging images or text out of the viewer is cancelled                                                                                   | Stops drag-to-desktop / drag-into-another-app exfiltration                                                                                                      |

The overlay sits above the toolbar and also covers the viewer in fullscreen.

## It does not touch copy, print, or download

`contentProtection` deliberately leaves text **copy/selection**, **printing**, and **downloading** alone — those already have their own flags, and bundling them here would hide that control. To lock them down, set the existing flags:

```vue
<DocumentViewer
  source="/confidential.pdf"
  wasm-url="/pdfium.wasm"
  :features="{
    contentProtection: true,
    selection: false, // no text selection / copy
    print: false, // no Print button / native print path
    download: false, // no Download button
  }"
/>
```

See [Features](/docs/document-viewer/features) for those flags.

## Notes

- **Runtime-toggleable.** Unlike most feature flags, `contentProtection` is read live — flipping it on or off on a mounted viewer takes effect without a `:key` remount.
- **Programmatic API is not gated.** The `DocumentViewerInstance` methods (annotation CRUD, `saveAsCopy`, etc.) are unaffected; this flag only changes in-page interaction behaviour.
- **No configuration object.** It's a single boolean by design. If you need to tune individual deterrents or the overlay copy, open an issue.
