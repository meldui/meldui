`DocumentViewer` is built on EmbedPDF's plugin architecture. Each plugin handles one concern (zoom, search, annotations, etc.) and is registered only when its corresponding `features` flag is on. This page lists every plugin in scope and links to its reference.

## Always-on (no feature flag)

These five plugins constitute the **minimum** PDF rendering pipeline. They're always registered:

- **[document-manager](/docs/document-viewer/plugins/document-manager)** ‚Äî orchestrates document lifecycle (load, switch, unload)
- **[viewport](/docs/document-viewer/plugins/viewport)** ‚Äî owns the scrollable container DOM
- **[scroll](/docs/document-viewer/plugins/scroll)** ‚Äî page-aware virtual scrolling
- **[render](/docs/document-viewer/plugins/render)** ‚Äî paints pages to canvas
- **[tiling](/docs/document-viewer/plugins/tiling)** ‚Äî splits large pages into tiles for memory efficiency

## View controls

Toggle viewport behaviour and orientation. Each opt-in via a `features` flag:

- **[zoom](/docs/document-viewer/plugins/zoom)** ‚Äî `features.zoom`
- **[rotate](/docs/document-viewer/plugins/rotate)** ‚Äî `features.rotate`
- **[spread](/docs/document-viewer/plugins/spread)** ‚Äî `features.spread`
- **[pan](/docs/document-viewer/plugins/pan)** ‚Äî `features.pan`
- **[fullscreen](/docs/document-viewer/plugins/fullscreen)** ‚Äî `features.fullscreen`

## Plugin registration pattern

Internally, `pluginRegistry.ts` is a pure function from `ViewerFeatures` ‚Üí `PluginRegistration[]`. The order matters because some plugins depend on others (e.g. `annotation` needs `interaction-manager`). The registry handles dependency ordering automatically; you don't need to think about it when toggling `features`.

```ts
import { resolveFeatures, DEFAULT_FEATURES } from '@meldui/vue'

const merged = resolveFeatures(DEFAULT_FEATURES, userFeatures)
// merged is the effective ViewerFeatures with auto-enabled dependencies
```
